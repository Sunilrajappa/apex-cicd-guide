If you've never used Git before

Five minutes of vocabulary that makes the rest of this guide make sense. Skip this chapter only if you're already comfortable committing and pushing.

What Git actually is

Git is a program that runs on your own computer and keeps a complete history of every change made to a folder of files. Think of it as an extremely detailed "track changes" for an entire project, not just one document. Every time you save a snapshot (called a commit), Git remembers exactly what every file looked like at that moment, forever, so you can always go back.

What GitHub actually is

GitHub is a website that stores a copy of your Git history online. Git itself doesn't need the internet — it works perfectly on your laptop alone. GitHub is just a remote backup and sharing point, and in this guide, it's also the trigger for our automation: GitHub watches for new commits and runs our deployment whenever one arrives.

TermWhat it means in plain language
Repository (repo)A project folder that Git is keeping history for. "Create a repo" just means "create that project folder, on GitHub."
CloneDownload a copy of a GitHub repo onto your own computer, with its full history attached.
CommitSave a snapshot of your current changes, with a short message describing what changed.
PushUpload your local commits to GitHub, so the online copy matches your computer.
PullDownload any new commits from GitHub that aren't on your computer yet.
BranchA separate line of work that doesn't affect the main project until it's merged in. Useful for trying something without risk.
Remote / originThe address of the GitHub copy that your local Git is linked to. "origin" is just the default nickname for it.
The only loop you actually need to remember
Almost everything in this guide reduces to three commands, run in this order, every single time you want to save and share a change: 
git add .
git commit -m "describe what you changed"
git push origin main
add stages your changes, commit saves a snapshot with a message, push sends it to GitHub. You will type some version of these three lines dozens of times — memorising this rhythm is worth more than memorising anything else in this guide.

The terminal, briefly

Why we keep opening a black window with white text, and the handful of commands that get reused constantly.

The Terminal app (search for "Terminal" in Spotlight), or the integrated terminal inside VS Code via View → Terminal, The PowerShell or Command Prompt app (search "PowerShell" in the Start menu), or the integrated terminal inside VS Code via View → Terminal, is a way of typing commands directly to your computer instead of clicking buttons. Git, and the Oracle tool we use to talk to the database (SQLcl), are both command-line tools — they don't have a graphical window, only this text interface. It looks intimidating at first but you'll only ever need a small, repeated set of commands.

PowerShell understands many Mac/Linux command names too (ls, cat, pwd) as built-in aliases, so most commands in this guide will work either way. Where they genuinely differ, both are shown.

A password with an @ symbol will break commands later
Throughout this guide, database connection commands are built as one long string like user/password@service. The terminal and Oracle's tools both treat @ as a meaningful separator character. If your password contains an @, the command will silently misread where the password ends — choose passwords with symbols like # instead, never @.
01

The mental model

Before touching a terminal, hold onto these three facts — everything else in this guide is just mechanics around them.

APEX and Git live in different worlds

A change made in App Builder exists only inside the Oracle database. Git cannot see it, version it, or move it anywhere, until it has been exported to plain .sql files on disk. Exporting is the bridge between the two worlds — every cycle in this guide starts there.

Two workspaces, one database

DEV and PROD are not two different servers — they are two isolated workspaces (and two isolated schemas) sitting on the same Always Free Autonomous Database. This is what makes the whole exercise free and realistic at the same time.

Three manual steps, the rest is automatic

Export, commit, push — that's the entirety of human involvement after setup is complete. GitHub Actions does the rest: downloading SQLcl, unpacking the wallet, setting workspace context, and running the import.

04

Git & GitHub repository

Create the empty repository first, then clone it locally — this avoids remote-mismatch issues entirely.

On GitHub

Go to github.com/new and create a repository:

  • Name it something memorable, e.g. apextogo
  • Leave README, .gitignore, and license unchecked
  • Click Create repository
On your Mac — clone the empty repo 
cd ~/Projects
git clone https://github.com/YOUR-USERNAME/apextogo.git
cd apextogo
On Windows (PowerShell) — clone the empty repo 
cd C:\Projects
git clone https://github.com/YOUR-USERNAME/apextogo.git
cd apextogo

If C:\Projects doesn't exist yet, create it first with mkdir C:\Projects.

Cloning instead of git init + git remote add means the remote is configured correctly from the very first command — no "remote origin already exists" errors later.

First push will ask you to sign in — this is expected
The first time you run git push, VS Code or your terminal will show a popup: "The extension 'GitHub' wants to sign in using GitHub." Click Allow. A browser tab opens to GitHub's login page; sign in and click Authorize. Return to your terminal — the push completes on its own. This authentication only happens once; every push afterward is silent.
05

Create the Autonomous Database

This is the one database that will host both your DEV and PROD workspaces.

If you haven't used Oracle Cloud before: this step asks Oracle to provision a real, dedicated database for you, hosted in one of their data centers, for free, permanently (within Always Free limits). It takes about two minutes to create and requires no database administration knowledge — Oracle manages patching, backups, and scaling automatically.

Choose the right workload type
The APEX workload type does not support wallet downloads or direct SQLcl connections on Free Tier — this blocks CI/CD entirely. Always choose Transaction Processing instead; APEX is enabled separately afterward.
  1. Open Autonomous Database in OCI
    After signing in to cloud.oracle.com, click the ☰ hamburger menu (top left) → Oracle Database → Autonomous Database → Create Autonomous Database.
  2. Configure the instance
    Display name and database name: anything memorable, they can be the same. Workload type: Transaction Processing. Deployment type: Serverless (the default). Toggle Always Free on — this is the switch that guarantees you'll never be billed for this database. Set and carefully record an ADMIN password somewhere safe, like a password manager or a note — you'll need it repeatedly.
  3. Avoid the @ symbol in any password you set here
    SQLcl's connection string uses @ as a separator between the password and the database address. A password containing @ will silently truncate the connection string and produce a confusing ORA-12154 "alias not found" error that looks unrelated to the real cause. Use # or other symbols instead.
  4. Click Create Autonomous Database, then wait
    You'll be returned to a database list showing a status of Provisioning. Refresh the page after a minute or two — it will change to Available with a green indicator. This means the database is running and ready.
  5. Enable APEX
    Click into the database you just created → Tools tab → find Oracle APEX in the list → click Open APEX. This is a one-time activation; APEX now runs inside this database, ready for workspaces to be created in the next chapter.
If the database shows "Stopped" later
Always Free databases pause automatically after a period of inactivity to save resources — this is normal and no data is lost. To resume: open the database page, click the ⋯ menu or the Start button, and wait about a minute for it to return to Available.
06

Create DEV and PROD workspaces

Two isolated workspaces, two isolated schemas, same database.

A workspace in APEX is a self-contained area with its own users, applications, and data — like having two separate APEX accounts on the same underlying database. A schema is the actual storage area inside the database where that workspace's tables live. We need two of each: one pair we'll experiment freely in (DEV), and one pair that represents "what's actually live" (PROD).

  1. Sign in to APEX Administration
    From the database's APEX URL, look for an Administration link (often at the bottom of the workspace sign-in page) or browse to https://your-host/ords/apex_admin. Sign in with username ADMIN and the ADMIN password you set when creating the database.
  2. Open Manage Workspaces
    In the top navigation, click Manage Workspaces → Create Workspace.
  3. Fill in the DEV workspace
    Workspace name SUNIL_DEV; when asked for a schema, choose create new schema and name it SUNIL_GDS with its own password (again, no @); admin username SUNILDEV; provide an email address (required, but doesn't need to be verified for this purpose). Click Create Workspace.
  4. Repeat for PROD
    Click Create Workspace again. This time: workspace name SUNIL_PROD, new schema SUNIL_GDS_PRD, admin username SUNILPRD.
FieldDEVPROD
WorkspaceSUNIL_DEVSUNIL_PROD
Schema (new)SUNIL_GDSSUNIL_GDS_PRD
Admin userSUNILDEVSUNILPRD
Why different schema names matter
Each schema is a fully isolated container of tables and data. Reusing the same schema name for both workspaces would mean DEV and PROD share live data — defeating the entire purpose of separating them.

Wallet & local connection

The wallet is the security credential that lets SQLcl — running on your computer, and later inside GitHub Actions — connect securely to the Autonomous Database.

A wallet is a small bundle of certificate files that proves to Oracle's servers that a connection attempt is legitimate, similar to how a key proves you're allowed into a building. Oracle Autonomous Database requires this extra layer of security by default — a username and password alone aren't enough. You'll download this bundle once, and reuse it both locally and (later) inside the automated pipeline.

  1. Download the wallet
    Database page → Database Connection → Download Wallet. Set a wallet password and save the zip somewhere stable — e.g. ~/Documents/wallet/ e.g. C:\Users\YourName\Documents\wallet\.
  2. Unzip it
    unzip Wallet_XXXXXXXX.zip -d wallet_unzipped
cat wallet_unzipped/tnsnames.ora
    Expand-Archive Wallet_XXXXXXXX.zip -DestinationPath wallet_unzipped
type wallet_unzipped\tnsnames.ora

    Alternatively, right-click the zip in File Explorer → Extract All works just as well.

    Note one of the service names, e.g. xxxxx_tp — this is what you connect to.
  3. Test the connection
    sql "admin/YourPassword123#@xxxxx_tp?TNS_ADMIN=/full/path/to/wallet_unzipped"
    sql "admin/YourPassword123#@xxxxx_tp?TNS_ADMIN=C:\full\path\to\wallet_unzipped"

    Use the same double-quoted form in both PowerShell and Command Prompt — it works in either.

    A SQL> prompt confirms success. This single command proves your computer can reach Oracle Cloud — GitHub Actions will run the exact same kind of command later, just from a different machine.
zsh and special characters
On zsh (the default Mac shell), an unquoted ? in the connection string is interpreted as a glob pattern and fails with no matches found. Always wrap the whole connection string in double quotes.
Quoting in PowerShell
PowerShell generally handles the ? and @ characters fine inside double quotes, but if a command behaves unexpectedly, try running it from Command Prompt instead — it has fewer special characters to worry about.

GitHub Secrets

Credentials GitHub Actions needs, stored encrypted, never visible in logs.

Convert the wallet to base64 (macOS) 
base64 -i ~/Documents/wallet/Wallet_XXXXXXXX.zip | pbcopy

pbcopy sends the output straight to your clipboard — nothing is printed to the screen.

Convert the wallet to base64 (PowerShell) 
[Convert]::ToBase64String([IO.File]::ReadAllBytes("C:\Users\YourName\Documents\wallet\Wallet_XXXXXXXX.zip")) | Set-Clipboard

Set-Clipboard sends the output straight to your clipboard — nothing is printed to the screen. This command must be run in PowerShell, not Command Prompt.

In your repository: Settings → Secrets and variables → Actions → New repository secret. Add each of these:

Secret nameValue
DB_PROD_USERThe PROD schema name, e.g. SUNIL_GDS_PRD
DB_PROD_PASSThat schema's password — no @ symbol
DB_SERVICEThe TNS service name, e.g. xxxxx_tp
WALLET_ZIP_B64Pasted from clipboard in the step above

The deploy workflow

This is the file that turns a push into a deployment. It downloads SQLcl, reconstructs the wallet, sets the correct APEX workspace context, then runs the import.

This file is written in YAML, a plain-text format that uses indentation (spaces, never tabs) to show structure — similar to how paragraphs and bullet points show structure in a document. GitHub reads this file automatically whenever it lives at the exact path .github/workflows/deploy.yml inside a repository; no separate registration step is needed. The file describes one job: on every push to the main branch, spin up a temporary Linux computer, install some tools, and run our deployment commands on it.

The one non-obvious step
Running install.sql directly fails with ORA-20001: Package variable g_security_group_id must be set. APEX needs to know which workspace it's installing into before the import runs — this is done with apex_util.set_security_group_id, looked up from the workspace name.
.github/workflows/deploy.yml 
name: Deploy DEV export to PROD APEX Workspace

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - name: Set up Java
        uses: actions/setup-java@v4
        with:
          distribution: 'temurin'
          java-version: '17'

      - name: Download SQLcl
        run: |
          wget -q https://download.oracle.com/otn_software/java/sqldeveloper/sqlcl-latest.zip
          unzip -q sqlcl-latest.zip

      - name: Set up wallet
        run: |
          echo "${{ secrets.WALLET_ZIP_B64 }}" | base64 -d > wallet.zip
          unzip -q wallet.zip -d wallet

      - name: Deploy to PROD workspace
        run: |
          cat <<'EOF' > deploy_script.sql
          set define off
          set serveroutput on

          declare
            l_workspace_id number;
          begin
            select workspace_id into l_workspace_id
            from apex_workspaces
            where workspace = 'SUNIL_PROD';

            apex_util.set_security_group_id(l_workspace_id);
          end;
          /

          @apex/f100/install.sql

          exit
          EOF
          sqlcl/bin/sql "${{ secrets.DB_PROD_USER }}/${{ secrets.DB_PROD_PASS }}@${{ secrets.DB_SERVICE }}?TNS_ADMIN=wallet" @deploy_script.sql
YAML is indentation-sensitive
Every step under steps: must start with exactly the same indentation. A single misaligned space (5 instead of 6, for example) will fail the workflow before it even runs.

First export & deploy

Putting it all together for the very first time.

  1. Build something in DEV you
    Sign in to the DEV workspace, create an application via App Builder, note its application ID (e.g. 100).
  2. Export it with SQLcl you
    sql "SUNIL_GDS/YourPassword@xxxxx_tp?TNS_ADMIN=/path/to/wallet_unzipped"
    sql "SUNIL_GDS/YourPassword@xxxxx_tp?TNS_ADMIN=C:\path\to\wallet_unzipped"
    then inside the SQL prompt: 
    apex export -applicationid 100 -split -expOriginalIds -skipExportDate
exit
  3. Move the export into the repo you
    SQLcl writes f100/ to whatever directory you ran it from — move it into the project's apex/ folder. 
    mv f100 ~/Projects/apextogo/apex/
    Move-Item f100 C:\Projects\apextogo\apex\
  4. Commit and push you
    cd ~/Projects/apextogo
git add .
git commit -m "feat: initial export of app 100"
git push origin main
    cd C:\Projects\apextogo
git add .
git commit -m "feat: initial export of app 100"
git push origin main
  5. Watch it deploy automatic
    Open the repository's Actions tab. The workflow fires immediately, runs for roughly 30–60 seconds, and a green check confirms the app is now live in the PROD workspace.

Once live, this is the entire workflow for every future change — no setup steps repeat.

  1. Make the change in DEV
    Add a page, change a region, adjust validation — anything, normally, inside the DEV workspace App Builder.
  2. Re-run the same export command
    sql "SUNIL_GDS/YourPassword@xxxxx_tp?TNS_ADMIN=/path/to/wallet_unzipped"
    sql "SUNIL_GDS/YourPassword@xxxxx_tp?TNS_ADMIN=C:\path\to\wallet_unzipped"
    apex export -applicationid 100 -split -expOriginalIds -skipExportDate
exit
    This regenerates only the files that actually changed — Git will show a small, readable diff rather than a wall of unrelated noise.
  3. Overwrite the existing export folder
    rm -rf ~/Projects/apextogo/apex/f100
mv f100 ~/Projects/apextogo/apex/
    Remove-Item -Recurse -Force C:\Projects\apextogo\apex\f100
Move-Item f100 C:\Projects\apextogo\apex\
  4. Review the diff before committing
    In VS Code's Source Control panel (or GitLens), confirm the changed files match what you expect — this is the moment a code review would happen on a real team.
  5. Commit with a meaningful message
    git add .
git commit -m "feat: add customer search filter to page 5"
git push origin main
    Adopting a prefix convention (feat:, fix:, chore:) makes the project history self-documenting over time.
  6. CI/CD takes it from here automatic
    The push triggers the same workflow: SQLcl downloads, the wallet unpacks, the PROD workspace context is set, and install.sql re-imports the application — overwriting the previous version in PROD with the new one.
Working with a team — branch first
For anything beyond solo experimentation, do the export and edit on a branch (git checkout -b feature/search-filter), open a pull request into main, and let a teammate review the diff before merging. The deploy only fires on pushes to main, so PROD is never touched by work still under review.
Rolling back
Because every deployed version is a Git commit, reverting PROD is reverting code: git revert <commit> and push. The workflow redeploys the prior version exactly as it existed at that commit.

Troubleshooting log

Real errors hit while building this pipeline, and what fixed each one.

SymptomCauseFix
remote: Repository not found Pushed before the GitHub repo existed Create the repo on GitHub first, then push
zsh: no matches found Unquoted ? in connection string parsed as a shell glob Wrap the entire connection string in double quotes
ORA-12154: TNS alias not found A password containing @ truncated the connection string Reset the password to avoid @ entirely
ORA-01017: invalid credential Wrong username for the database type, e.g. assuming admin exists everywhere Connect via Docker as sys as sysdba and check dba_users for the real schema name
